1 # TimerManager -- Callback-Based Event Handling {#timer_manager}
2 
3 [TOC]
4 
5 In some places, OMNeT++ provides mechanisms to react to some sort of event, that are harder to use than they should.
6 It is perfectly possible to implement the reaction to the event, but it involves a lot of boilerplate code and no real abstraction.
7 
8 One example for this are self messages, which are often used to implement timers (one shot or recurring) for individual modules.
9 To implement a single timer, one typically has to:
10 
11 - define a class member to contain the pointer to the cMessage instance.
12 - create the cMessage instance
13 - schedule the cMessage instance
14 - detect the arrival of that particular cMessage, usually in some if-cascade in or called by handleMessage
15 - perform the actual action
16 - possibly re-schedule the message, if the event is recurring
17 - delete the cMessage when done and/or the object gets destroyed
18 
19 ```{.cpp}
20 // Example module with a one-shot timer firing 2.5 seconds after initialize and a recurring timer firing every 1 second
21 class WitoutTimers: public cSimpleModule {
22 protected:
23  // define class member to contain the pointer to the cMessage instance
24  cMessage* recurringTrigger;
25  cMessage* oneshotTrigger;
26  void initialize() override {
27  // create the cMessage instance
28  recurringTrigger = new cMessage("recurring trigger");
29  oneshotTrigger = new cMessage("one-shot trigger");
30  // schedule the cMessage instance
31  scheduleAt(1, recurringTrigger);
32  scheduleAt(2.5, oneshotTrigger);
33  // ...
34  }
35  void handleMessage(cMessage* msg) override {
36  if (msg->isSelfMessage()) {
37  // detect the arrival of a particular cMessage
38  if (msg == recurringTrigger) {
39  // perform actual action
40  std::cerr << "recurring Timer called from " << this->getFullPath() << std::endl;
41  // re-schedule the message
42  scheduleAt(1, recurringTrigger);
43  }
44  if (msg == oneshotTrigger) {
45  std::cerr << "one-shot timer fired from " << this->getFullPath() << " at " << simTime() << std::endl;
46  }
47  }
48  }
49 public:
50  virtual ~WithoutTimers() {
51  // delete the cMessage when the object gets destroyed
52  if(recurringTrigger) {
53  cancelAndDelete(recurringTrigger);
54  recurringTrigger = nullptr;
55  }
56  if(oneshotTrigger) {
57  cancelAndDelete(oneshotTrigger);
58  oneshotTrigger = nullptr;
59  }
60  }
61 };
62 ```
63 
64 This is highly repetitive and scatters code all over the class, which is bad.
65 Furthermore, it has to be repeated for every single timer.
66 
67 With the [TimerManager] suite, most of the boilerplate is abstracted away.
68 The [TimerManager](@ref veins::TimerManager) contains all of it and provides an interface with clear semantics.
69 Now the only steps left are:
70 
71 - per-module setup of the [TimerManager] instance (only necessary once per module):
72  - define and instantiate the [TimerManager]
73  - forward all self-messages to the [TimerManager] instance (in handleMessage)
74 - define the individual timers:
75  - encode the reaction to the timer firing in a functor, e.g., a lambda
76  - specify when and how often a timer shall fire by constructing a [TimerSpecification]
77  - register the timer with the [TimerManager] instance
78 
79 ```{.cpp}
80 // Example module with same functionalty as above
81 class WithTimers: public cSimpleModule {
82 protected:
83  veins::TimerManager timerManager(this); // define and instantiate the TimerManager
84  void handleMessage(cMessage* msg) override {
85  if (msg->isSelfMessage()) {
86  timerManager.handleMessage(msg); // forward all self-messages to the TimerManager instance
87  }
88  // ...
89  }
90  void initialize() override {
91  // encode the reaction to the timer firing with a lambda
92  auto recurringCallback = [this](){
93  std::cerr << "recurring Timer called from " << this->getFullPath() << std::endl;
94  };
95  auto oneshotCallback = [this](){
96  std::cerr << "one-shot timer fired from " << this->getFullPath() << " at " << simTime() << std::endl;
97  };
98  // specify when and how ofthen a timer shall fire
99  auto recurringTimerSpec = veins::TimerSpecification(recurringCallback).interval(1);
100  auto oneshotTimerSpec = veins::TimerSpecification(oneshotCallback).oneshotin(2.5);
101  // register the timer with the TimerManager instance
102  timerManager.create(recurringTimerSpec, "recurring timer");
103  timerManager.create(oneshotTimerSpec, "one-shot timer");
104  }
105 };
106 ```
107 
108 Note: the [TimerManager] works fine with self-messages not handled by it.
109 The return code of [TimerManager::handleMessage] indicates whether the message passed to it could be handled by a timer.
110 ```{.cpp}
111 // Alternative implementation for co-existence with other self-messages
112  void handleMessage(cMessage* msg) override {
113  if (msg->isSelfMessage() && timerManager.handleMessage(msg)) return;
114  // ...
115  }
116 ```
117 
118 All cleanup and re-scheduling chores are done by the [TimerManager]
119 As the [TimerManager] instance can be value member (and not a pointer to a heap-allocated object like cMessage), there is no need for a custom destructor.
120 
121 When using inheritance trees, the [TimerManager] can easily used by different levels (as long as it is a protected or public member or accessible otherwise).
122 Even multiple instances coule be used, as long as message delegation is correctly implemented.
